Universal Commerce Protocol overview

チャットの相談がチェックアウトになるまで

購入者は一直線に進むのではなく、AIと会話しながら、検索、比較、 条件変更、カート、チェックアウト準備を何度も行き来します。

AI Agentとの会話
会話のたびに、検索、詳細取得、カート更新、チェックアウト処理のtool呼び出しが起きます。
1

購入者がAIに相談

目的、予算、好み、配送先、画像などのシグナルが会話に入ります。

2

Agentがtoolを呼ぶ

MCPコネクターが会話の文脈をUCP向けのリクエストに変換します。

3

Catalogが候補を返す

商品、画像、価格、バリアント、マーチャント情報がデータとして返ります。

4

カートやチェックアウトを更新

商品、購入者情報、合計金額、次に必要な状態を準備します。

5

購入者が確認または再相談

承認、条件変更、別商品の比較などにより、再び1へ戻ります。

購入準備が整うまで、5から1へ何度も戻る。
Catalogは「何を買うべきか」に答える

AgentはShopify上のマーチャントを横断して検索し、配送先に合う商品や バリアントを、購入者が複数のストアを巡回する前に提示できます。

Checkoutは「どこで安全に支払うか」に答える

チェックアウトのルールと決済の信頼はマーチャント側に残ります。 Agentは入力補助を行い、確認や支払いが必要な時点で安全に引き継ぎます。

要点: UCPは、商品情報、チェックアウト方針、決済の信頼を マーチャント側に保ったまま、AIとの購買会話を反復型のコマース体験にします。

UCPの責任分担モデル

UCPでは4つの主体が購買の信頼を分担する

UCPは、1つのシステムにすべてを任せずにコマースを連携します。 Platformが会話を進め、Merchantが販売を担い、決済資格情報と 決済処理は別々の信頼境界に保たれます。

P

Platform

購入者の意図を理解し、toolを呼び、候補を提示して購買ジャーニーを 調整するAIホストまたはAgentクライアントです。

↔ UCP protocol
M

Merchant

商品情報、チェックアウト方針、フルフィルメント、リスク判断、 生成された注文に責任を持つ販売者です。

CP

Credential Provider

Shop Pay、Google Pay、Apple Payなど、購入者が承認した 決済資格情報を発行するウォレットです。

↔ AP2 / payment token
PSP

Payment Service Provider

Shopify PaymentsやStripeなど、Merchantのために取引を オーソリし、売上を精算する決済処理事業者です。

P→CP
Platformが資格情報を要求 Agentが再利用可能な決済秘密情報を受け取る前に、購入者がウォレットで承認します。
M→PSP
Merchantが決済を要求 オーソリ、不正検知、売上確定、精算はMerchantとPSPの信頼境界に残ります。
このサンプル: Shopify MCPをラップし、Platformと Merchantの間をつなぎます。Credential ProviderとPSPは実装しません。 対象範囲の詳細を見る。
Checkoutの状態 FAQ English version

Checkout status state machine

statusが次に行動する主体を決める

Checkout MCPのすべての応答にはstatusがあります。Agentは修正可能な 情報を更新し、escalationが必要なら購入者をcontinue_urlへ 引き継ぎ、Checkoutが準備完了の場合だけ完了処理を行います。

General access 多くのAgent統合では、requires_escalationとcontinue_urlが通常の引き継ぎ経路です。
Agentによる完了 現在の状態がready_for_completeで、使用可能な決済状態がある場合だけcomplete_checkoutを呼びます。
Terminal state completedとcanceledでセッションは終了します。購入者を引き継いだ後はget_checkoutで再確認します。
ルール: tool名だけから次のアクションを推測せず、返された statusとmessagesを読みます。詳しい状態モデルを見る。

実装FAQ

6つの違いを知れば実装の誤解を防げる

各項目を開くと、実装境界の要点を確認できます。保守される正本、詳しい説明、 Shopifyドキュメントへのリンクは tips.mdにあります。

Global Catalogのmetadataは標準UCPで、Agentが動的に変更できますか？

いいえ。dev.shopify.catalog.globalは、ML推論metadataなどを標準UCP Catalogへ加えるShopify extensionです。

公開インターフェースにmetadataの動的overrideはありません。Saved Catalogは結果を絞り込みますが、attributesやtop_featuresなどを書き換えません。

Shop Pay WalletとUCP Shop Pay handlerは同じ実装ですか？

いいえ。Shop Pay WalletはコマースのcheckoutへUIとpayment request sessionを加えます。UCP handlerはAgentのclient_idと1回限りのShop Tokenを使うdelegated paymentです。

どちらかを実装しても、もう一方が設定されたり利用可能になったりはしません。

ECPはRemote MCP serverだけで実装できますか？

いいえ。Agent hostがcontinue_urlをWebViewで開き、JSON-RPCメッセージを交換し、委譲されたnative UIを実装する必要があります。

MCP serverはURLを返せますが、任意のAI clientにこれらの機能を追加できません。

独自Catalogを作り、Catalog IDを渡す必要がありますか？

いいえ。Global Catalogはデフォルトで対象となるShopify Merchantを検索します。Saved Catalogは繰り返し使う範囲やfilterのための任意機能です。

使う場合はShopify GIDではなくsaved_catalog_slugを渡します。API credentialやproduct IDとは別の識別子です。

Catalogの検索結果や商品画像をcacheできますか？

いいえ。ShopifyはCatalog検索結果をcacheしないよう定め、商品画像は関連Merchantの商品情報とともにリアルタイム表示する必要があります。

product dataを再利用しないtokenやendpointのcacheは別です。

単一MerchantならGlobalとStorefrontのどちらを使いますか？

既知の1つのMerchantにはStorefront Catalogを使います。複数Merchantの商品発見、比較、推薦にはGlobal Catalogを使います。

どちらもUCP Catalogを実装しますが、範囲、認証、Shopify extensionのfieldが異なります。

最新情報: Shopifyのsurface変更時は、詳しい正本である tips.mdを参照してください。
